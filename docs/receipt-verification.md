# Receipt Verification

Every tool call produces a receipt with a unique `reference_id`. This page explains how to verify receipts, control what data they contain, and manage explorer visibility.

## `reference_id` vs `receipt_id`

Each receipt has two identifiers:

- **`reference_id`** — generated by the SDK on the client side. Available immediately in your code via `aevs.get_reference_ids()`, even before the receipt reaches the backend. This is the ID you work with programmatically.
- **`receipt_id`** — assigned by the AEVS backend when the receipt is stored. You will see it in the explorer and in API verification responses, but it is not available in the SDK.

Both can be used to search for receipts on the explorer.

## Verifying a receipt

### Via the Explorer

Go to [explorer.aevs.fetch.ai](https://explorer.aevs.fetch.ai) and search by `reference_id` or `receipt_id`. The explorer shows the full receipt details including tool name, timing, status, and (depending on visibility settings) the inputs and outputs.

### Via the API (no auth required)

```
GET https://api.aevs.fetch.ai/v1/receipts/verify/<reference_id>
```

Anyone with the `reference_id` can verify the receipt through the API — no API key needed. The response includes the receipt metadata, hash chain status, and KMS signature information.

### Public receipts API

The backend also provides public endpoints for browsing receipts from agents with explorer listing enabled:

| Method | Path | Description |
|--------|------|-------------|
| GET | `/v1/public/receipts` | List public receipts (supports pagination and filters) |
| GET | `/v1/public/receipts/stats` | System-wide statistics (total, public, private, proof_only, success, error counts) |
| GET | `/v1/public/receipts/lookup?q=<id>` | Look up by `receipt_id` or `reference_id` |
| GET | `/v1/public/receipts/{receipt_id}` | Full detail for a specific receipt |
| GET | `/v1/public/receipts/{receipt_id}/verify-kms` | Verify KMS signature (ECDSA P-256) |

These endpoints require no authentication. They only return receipts from agents where `receipts_public` is enabled on the dashboard.

### KMS verification

In production, every ingested receipt is additionally signed by a GCP Cloud KMS HSM key (ECDSA P-256). The `/verify-kms` endpoint performs cryptographic verification of this signature and returns the result, allowing independent proof that the receipt was stored by the AEVS backend.

### Getting reference IDs from your code

```python
refs = aevs.get_reference_ids(clear=True)
for ref in refs:
    print(f"Tool: {ref['tool_name']}, Ref: {ref['reference_id']}")
```

Each entry in the list contains:

| Field | Description |
|-------|-------------|
| `seq` | Receipt sequence number |
| `tool_name` | Name of the tool |
| `reference_id` | UUID to verify this receipt |
| `run_id` | Framework-assigned run ID |
| `tool_call_id` | Tool call ID from the framework |

You can also look up a specific `reference_id` by its framework `run_id` or `tool_call_id`:

```python
ref_id = aevs.get_reference_id("framework-run-id-here")
# Returns the reference_id string, or None if not found
```

## Two levels of visibility

Receipt visibility is controlled at **two levels** that work together:

### 1. SDK level — `receipt_visibility`

Set in your code when you call `aevs.configure()`. Controls what data is *included* in each receipt.

### 2. Dashboard level — explorer listing

Toggle per agent at [explorer.aevs.fetch.ai/dashboard/agents](https://explorer.aevs.fetch.ai/dashboard/agents). Controls whether the agent's receipts are *listed on the public explorer*.

When explorer listing is off, receipts are not browsable on the public explorer. However, the **verify API still works** if you have the `reference_id` — so programmatic verification is always possible.

## SDK visibility modes

### `"public"` — open audit

```python
aevs.configure(api_key=..., agent_id=..., receipt_visibility="public")
```

- **What is stored:** Full inputs, outputs, metadata, and hash chain
- **Who sees payloads:** Anyone with the `reference_id` or explorer access
- **Best for:** Demos, shared audits, or workflows where reviewers need the full picture

### `"private"` (default) — prove the call, hide payloads

```python
aevs.configure(api_key=..., agent_id=..., receipt_visibility="private")
```

- **What is stored:** Full inputs and outputs in the backend (your team can view them)
- **Who sees payloads:** Only your account. Public verify and explorer show `[REDACTED]` for inputs and outputs.
- **Still public:** Tool name, status, timestamps, signatures, and hash chain. Auditors can confirm execution without seeing sensitive fields.

### `"proof_only"` — strictest

```python
aevs.configure(api_key=..., agent_id=..., receipt_visibility="proof_only")
```

- **What is stored:** Inputs and outputs are stripped (set to `null`) before the receipt is submitted. Only metadata (tool name, timing, status), signatures, and chain data are kept.
- **Who sees payloads:** No one. There is no payload to retrieve, even for the owner.
- **Best for:** PCI, health, or other regulated data you must not persist, while still proving the tool ran.

### Comparison

| | `public` | `private` | `proof_only` |
|---|---|---|---|
| Inputs & outputs stored | Full | Full (team-only access) | Stripped (`null`) |
| Public verify/explorer shows | Full payloads | `[REDACTED]` | `null` |
| Tool name, status, timing | Yes | Yes | Yes |
| HMAC & hash chain | Yes | Yes | Yes |

You can also set this via the `AEVS_RECEIPT_VISIBILITY` environment variable.

## What verification proves

When you verify a receipt, you confirm:

1. **Authenticity** — the receipt was signed by the SDK with a valid API key
2. **Integrity** — the receipt has not been modified since it was created
3. **Ordering** — the receipt's position in the hash chain is valid (it follows the correct previous receipt)
4. **Timing** — when the tool call started and ended
5. **Chain status** — the backend reports whether the receipt is `anchor`, `linked`, `mismatch`, `gap`, `broken`, or `unverified` in its session chain

Verification does **not** prove that the tool actually performed the claimed action on an external system. It proves the SDK recorded the call and its result as reported by the tool.

### `proof_only` hashes

When `receipt_visibility="proof_only"`, inputs and outputs are stripped from the receipt before submission. However, the SDK computes `input_hash` and `output_hash` (SHA-256 of canonical JSON) and includes them in the receipt. This allows you to later prove that specific data was processed without the data ever leaving your host.
